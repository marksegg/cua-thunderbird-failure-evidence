## CHALLENGE 4: IS THE CONDITION ORDER A VERIFIER BUG?

### Finding: YES. This is an independent verifier bug.

**Instruction order:**
> "with TWO conditions: **From contains 'billing@'** AND **Subject contains 'Invoice'**"

Order: From first, Subject second.

**Model behavior (Step 45, Step 54):**
> Step 45: "I need to change the first condition to 'From' 'contains' 'billing@'"
> Step 54: "Both conditions are set: 1. From contains 'billing@' 2. Subject contains 'Invoice'"

The model followed the instruction order exactly.

**Verifier expected conditions:**
```json
"condition": [
    "AND (subject,contains,Invoice)",
    "AND (from,contains,billing@)"
]
```

Order: Subject first, From second. **This CONTRADICTS the instruction.**

**The `_match_record` function uses strict list equality:**
```python
def _match_record(pattern, item):
    return all(k in item and item[k] == val for k, val in pattern.items())
```

Python list equality is ORDER-SENSITIVE:
```python
["AND (subject,contains,Invoice)", "AND (from,contains,billing@)"] 
!= 
["AND (from,contains,billing@)", "AND (subject,contains,Invoice)"]
```

**Compare with Ironbridge verifier:**
```json
"condition": [
    "AND (from,contains,notifications@fedexfreight.com)",
    "AND (subject,contains,delivered)"
]
```
Ironbridge instruction: "From contains... AND Subject contains... (add the rules in this order)"
Ironbridge verifier: From first, Subject second -- **matches the instruction.**

**Verdict:** The Vendor Management verifier has the conditions in the WRONG ORDER relative to its own instruction. Even if the model created the filter under Local Folders, it would STILL fail because it followed the instruction order (From first) while the verifier expects Subject first. This is an unambiguous verifier bug.
